## 1. Google XYZ Formula

**Formula:** "Accomplished [X] as measured by [Y] by doing [Z]"
— Laszlo Bock, former Google SVP of People Operations

- **X** = accomplishment (strong action verb)
- **Y** = quantifiable metric (%, $, users, time)
- **Z** = method/approach (tools, technical strategy)
- Order can vary — lead with the metric if impressive
- First 1-2 bullets per role should have all three elements; remaining bullets need at least Action + Context
- Keep each bullet to 1-2 lines max

## 2. FAANG Resume Best Practices (2025-2026)

- One page maximum, single-column, reverse-chronological
- Recruiters spend 6-10 seconds on initial scan
- Quantified achievements get 40% more interview requests
- Mirror job description keywords in Skills and bullets
- Only list skills with genuine professional experience
- Include 2+ projects with GitHub/live demo links
- Tailor resume per application — highest-ROI activity

**Sources:**
- https://www.techinterviewhandbook.org/resume/
- https://www.designgurus.io/blog/best-resume-formats-for-faang-and-top-tech-companies-2025
- https://igotanoffer.com/blogs/tech/tech-resume-examples

## 3. ATS Optimization

- 99.7% of recruiters use keyword filters in ATS (Jobscan 2025)
- 75%+ of companies use ATS to filter before human review
- Standard section headers: "Experience", "Skills", "Education"
- No tables, columns, graphics, headers/footers, text boxes
- Use keywords exactly as in JD; include both acronym and full form
- Target 65-75% keyword match rate
- .docx for ATS submission, PDF for direct sends
- Test with ATS simulation tools when possible

**Sources:**
- https://blog.theinterviewguys.com/ats-resume-optimization/
- https://www.jobscan.co/blog/20-ats-friendly-resume-templates/

## 4. Ex-FAANG Recruiter Tips

- Lead with years of role-relevant experience
- Focus on impact, not responsibilities or technologies
- Avoid "word salad" — one key point per bullet
- One page, black-and-white, clean layout
- Create distinct resumes for each role type
- Include: name, email, phone, city, LinkedIn
- Proofread ruthlessly — typos are immediate red flags

## 5. STAR Method for Engineering Resumes

**STAR** = Situation, Task, Action, Result
On resumes, compress into one bullet — don't label each part.

**Engineering Examples:**
- Frontend: "Reduced checkout page load time by 60% by implementing lazy loading and code splitting, increasing conversion rate by 12% ($2M annual revenue impact)"
- Backend: "Led migration of payment processing from monolith to microservices, reducing failed transactions by 35% and handling 2M+ daily transactions across 12 countries"
- Reliability: "Identified and resolved critical memory leak in production, reducing crash rate by 90% and improving app store rating from 3.2 to 4.6 stars"
- Platform: "Architected React component library used by 15 engineers, reducing UI development time by 40% and ensuring consistent design across 50+ pages"

**Sources:**
- https://resumegenius.com/blog/resume-help/star-method-resume
- https://www.careereducation.columbia.edu/resources/resumes-impact-creating-strong-bullet-points

## 6. Cross-Cutting Themes

These patterns emerged consistently across ALL credible sources:

1. **Impact over duties** — every bullet must answer "so what?" with a number
2. **One page, single column, standard fonts** — universal across FAANG/ATS/recruiter advice
3. **Tailor per application** — match keywords to JD; highest-leverage activity
4. **XYZ is the gold standard** for bullet points
5. **First bullet of most recent role is the most important line** — optimize it ruthlessly
6. **Spell out acronyms AND include abbreviations** for both humans and ATS
7. **Proofread** — typos are an automatic rejection signal at top companies

## 7. Professional Summary — When to Include

Industry opinions are split:

- **Google official stance:** No summary needed — the resume IS the summary (per IGotAnOffer)
- **Tech Interview Handbook (ex-Meta engineer):** Strongly recommends it — interviewers don't read details
- **Enhancv:** Senior engineers can skip it

**Recommended rules:**
- Include if the user's background needs context: career switch, non-linear path, returning after a gap, or transitioning between domains
- Skip if the background is straightforward (consistent roles in the same domain)
- If included: must contain at least one quantified number
- Never exceed 3 lines
- May reference 1-2 strongest metrics from bullets as hooks, but never copy full bullet descriptions verbatim

**Sources:**
- https://igotanoffer.com/blogs/tech/google-resume-examples-tips
- https://www.techinterviewhandbook.org/resume/
- https://enhancv.com/resume-examples/google/

## 8. Common Over-Inflation Patterns to Avoid

These patterns are especially prevalent in AI-assisted resume writing. Watch for and correct them:

### 1. Role Inflation
Writing "Led" when the person coordinated or contributed.
- Fix: Match verb to actual role. "Coordinated" and "Drove" are strong verbs too.

### 2. Technical Inflation
User used a tool → written as "Architected the system using..."
- Fix: Describe what was actually built, not the grandest possible framing.

### 3. Number Inflation
Team-level or company-level metrics attributed to the individual.
- Fix: Only use numbers the user directly influenced. A web developer's bullet shouldn't cite app download counts.

### 4. Terminology Inflation
Simple features packaged with grand vocabulary.
- Bad: "Architected a dynamic page orchestration platform" (it was a JSON config feature)
- Good: "Built a JSON-driven configuration system enabling non-engineers to create pages without code"

### 5. Skills Section Inflation
Listing every technology ever encountered, including one-time tutorials.
- Fix: Only list technologies with genuine professional experience.

### 6. Association Inflation
Using impressive numbers that the user was near but didn't directly impact.
- Bad: Web developer citing "platform serving 10M users" when they built an internal admin tool
- Good: "Built internal admin dashboard used by 50 operations staff daily"

**Self-check rule:** After every bullet, ask: "If an interviewer drills into this for 2 minutes, can the user speak fluently and confidently?" If not, the wording is too inflated.
